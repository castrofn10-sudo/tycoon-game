2. EVENT CATEGORIES
Classify events into categories:

A) NEGATIVE_PR
Examples:
- false advertising
- public backlash
- trust loss

Trigger only if:
- hype was too high compared to finalScore
- bugs are medium/severe
- player ignored pre-launch improvements
- reputation is already fragile

B) POSITIVE_PR
Examples:
- surprise success
- strong reviews
- viral popularity

Trigger only if:
- finalScore is high
- bugs are low/none
- hype and delivery are aligned
- sales are strong

C) TECH
Examples:
- technical criticism
- compatibility issues
- hardware bottlenecks

Trigger only if:
- bugs are present
- poor hardware fit exists
- rushed launch happened

D) MARKET
Examples:
- market slowdown
- trend shift
- competitor pressure
- genre boom

Trigger only if:
- current year/era supports it
- market state justifies it
- competition exists

E) INTERNAL
Examples:
- team fatigue
- production friction
- management strain

Trigger only if:
- workload is high
- too many projects are active
- company expansion is under pressure

- A bug-free high-quality game must NOT receive bug-style negative events

- Low hype should block false advertising style events
- Positive and negative event types should not conflict in the same launch moment unless explicitly justified

4. COOLDOWNS
Prevent spam:
- Same event cannot repeat for 12–24 in-game months

- Major negative events should be rare
