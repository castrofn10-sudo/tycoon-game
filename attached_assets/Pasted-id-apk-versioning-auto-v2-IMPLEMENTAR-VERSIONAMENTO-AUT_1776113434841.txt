IMPLEMENTAR VERSIONAMENTO AUTOMÁTICO DE BUILD APK

OBJETIVO:
Configurar apenas o versionamento do app para que cada novo build gere uma nova versão e um novo versionCode, sem alterar qualquer sistema do jogo.

REGRAS ABSOLUTAS:

- NÃO modificar gameplay
- NÃO modificar IA
- NÃO modificar eventos
- NÃO modificar economia
- NÃO modificar UI
- NÃO modificar saves
- NÃO modificar lógica do app
- NÃO alterar dependências
- NÃO refatorar arquivos fora de configuração

---

[1] ESCOPO PERMITIDO

Permitir alterações SOMENTE em arquivos de configuração de build, como:

- app.json
- app.config.js / app.config.ts
- eas.json
- scripts de build/versionamento, se estritamente necessário

NÃO tocar em:

- app/
- components/
- context/
- hooks/
- constants/ (exceto se houver arquivo já existente apenas de versão)
- gameEngine
- telas
- sistemas do jogo

---

[2] VERSIONAMENTO AUTOMÁTICO

Configurar para que a cada novo build:

- version seja incrementada automaticamente no PATCH
- android.versionCode seja incrementado automaticamente em +1

Exemplo:
Antes:
version: 1.0.0
versionCode: 1

Depois:
version: 1.0.1
versionCode: 2

Depois:
version: 1.0.2
versionCode: 3

---

[3] PADRÃO DE VERSIONAMENTO

Usar semantic versioning:

MAJOR.MINOR.PATCH

Regras:

- MAJOR permanece fixo, salvo alteração manual futura
- MINOR permanece fixo, salvo alteração manual futura
- PATCH incrementa automaticamente a cada build

NÃO alterar MAJOR ou MINOR automaticamente.

---

[4] ANDROID VERSION CODE

Garantir que:

- versionCode sempre aumente +1 por build
- nunca repita valor anterior
- nunca retroceda
- seja compatível com geração de APK/AAB no Android

---

[5] NOVO BUILD / NOVO ARTEFATO

Garantir que cada build resulte em:

- nova versão do app
- novo versionCode
- novo artefato de build

Importante:

- NÃO prometer controle do nome final do arquivo baixado se isso depender da plataforma de build
- apenas garantir que o build gerado seja tratado como uma nova versão instalada/baixada

---

[6] IMPLEMENTAÇÃO SEGURA

Prioridade de implementação:

1. Usar configuração nativa do Expo/EAS se suportado
2. Se necessário, criar script simples e isolado de versionamento
3. Evitar qualquer automação complexa

Se precisar criar script:

- ele deve apenas ler a versão atual
- incrementar PATCH
- incrementar versionCode
- salvar de volta nos arquivos de configuração
- não tocar em mais nada

---

[7] PROIBIÇÕES

NÃO:

- criar sistema de CI/CD complexo
- mexer em runtime do app
- alterar lógica de download
- alterar nome de telas
- mexer em engine
- alterar pacote do app
- mudar identificadores do projeto sem necessidade

---

[8] VALIDAÇÃO FINAL

Ao terminar, informar apenas:

- quais arquivos de configuração foram alterados
- como o PATCH é incrementado
- como o versionCode é incrementado
- confirmar que nenhum sistema do jogo foi modificado

Se houver limitação da plataforma para “novo nome de arquivo de download”, explicar isso sem tentar contornar mexendo em partes não relacionadas.

---

[9] RESULTADO ESPERADO

A cada novo build:

- versão do app sobe no PATCH
- versionCode sobe +1
- o build é reconhecido como novo
- sem impacto no jogo
